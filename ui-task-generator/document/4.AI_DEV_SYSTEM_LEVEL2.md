# 🤖 AI DEV SYSTEM — LEVEL 2

## 🎯 Mục tiêu
Level 2 nâng cấp hệ thống AI Dev Assistant từ “đọc task rồi code” thành “phân tích yêu cầu sâu hơn, tự tìm file liên quan, tự sinh checklist triển khai, tự review và đề xuất test”.

Hệ thống này giúp Claude Code làm việc gần với một senior developer hơn:
- hiểu project trước khi code
- tự định vị module liên quan
- phân tích ảnh/UI thành task kỹ thuật
- sinh kế hoạch triển khai chi tiết
- tự review thay đổi
- tự đề xuất test cases và risk cases

# 🚀 3. Flow chuẩn Level 2

## Bước 1 — Intake
Nguồn đầu vào có thể là:
- task file `.ts`
- mô tả text
- ảnh giao diện
- bug report
- yêu cầu thêm field/module

Claude phải phân loại task thành 1 trong các loại:
- `feature`
- `bugfix`
- `refactor`
- `enhancement`
- `test-only`
- `ui-to-code`

---

## Bước 2 — Project Mapping
Claude phải làm các việc sau trước khi code:

1. Xác định module liên quan
2. Xác định layer liên quan:
   - validator
   - repository
   - service
   - route
   - admin ui
   - site ui
3. Liệt kê file cần đọc trước
4. Xác định module reference gần nhất
5. Dự đoán file cần sửa
6. Chỉ ra risk nếu sửa sai phạm vi

---

## Bước 3 — Analysis Output
Trước khi code, Claude phải luôn xuất ra:

### A. Task Summary
- mục tiêu task
- phạm vi module
- loại task
- đầu vào/đầu ra

### B. File Analysis
- files bắt buộc phải đọc
- files có thể cần sửa
- files không nên đụng vào

### C. Implementation Plan
- bước 1
- bước 2
- bước 3
- rollback/risk strategy

### D. Risks
- risk về UI
- risk về API
- risk về DB
- risk về backward compatibility

---

## Bước 4 — Controlled Implementation
Khi được phép code, Claude phải tuân thủ:

- chỉ sửa file liên quan trực tiếp
- không refactor lan man
- không thêm thư viện mới nếu không cần
- không phá naming conventions
- không bỏ qua type errors
- không đổi output format nếu task không yêu cầu

---

## Bước 5 — Self Review
Sau khi code, Claude phải tự review theo 4 nhóm:

### 1. Architecture
- đúng 3-layer chưa
- route có bị nhồi logic không
- service có validation chưa
- repository có over-fetching không

### 2. Type Safety
- có any không
- infer type từ zod chưa
- null/undefined có được xử lý chưa

### 3. Scope
- có sửa ngoài task không
- có thay đổi side effects không

### 4. UX/Runtime
- submit flow có ổn không
- router.refresh có cần không
- loading/error state có đúng không

---

## Bước 6 — Testing Design
Claude phải sinh:
- test cases bắt buộc
- edge cases
- regression cases
- manual QA checklist

# 🛠️ 5. Chế độ hoạt động

## Mode A — Task to Code
Dùng khi đã có task file.

Flow:
- đọc task
- phân tích
- lên plan
- code
- review
- test

## Mode B — UI to Task
Dùng khi bà chủ chỉ có:
- ảnh
- bản vẽ
- ghi chú

Flow:
- phân tích UI
- convert thành task file
- xác định module
- rồi mới code

## Mode C — Bugfix
Dùng khi có bug report.

Flow:
- tái hiện bug
- tìm root cause
- xác định file liên quan
- đề xuất fix nhỏ nhất
- test regression

## Mode D — Safe Refactor
Dùng khi cần cải thiện code nhưng không đổi hành vi.

Flow:
- xác định pain points
- chỉ refactor trong file/module nhỏ
- không đổi output
- tạo regression checklist

# 🔍 8. Auto Detect Checklist

Khi làm task, Claude phải tự hỏi:

- task này có đụng DB không?
- task này có thêm field không?
- task này có làm thay đổi response không?
- task này có ảnh hưởng list page không?
- task này có cần update form không?
- task này có cần update table/filter không?
- task này có cần phân quyền không?
- task này có cần migration không?

Nếu câu trả lời là có, phải nêu ra trước khi code.

# 🧱 9. Scope Control System

Đây là phần quan trọng nhất của Level 2.

Claude phải chia file thành 3 nhóm:

## Nhóm 1 — Bắt buộc đọc
Files chắc chắn phải đọc trước khi sửa.

## Nhóm 2 — Có thể sửa
Files có khả năng chỉnh sửa trực tiếp.

## Nhóm 3 — Không được đụng
Files ngoài phạm vi, trừ khi task yêu cầu rõ.

Ví dụ:
- task menu filter thì không nên đụng `product.service.ts`
- task UI admin thì không nên refactor `site/` nếu không cần

# 🧠 10. Hệ prompt dùng hằng ngày

## Prompt 1 — Phân tích task
```text
Read skill + task. Classify task type, identify module, list files to read, summarize current logic, write implementation plan and risks. Do not code yet.
```

## Prompt 2 — Thực thi
```text
Proceed with implementation. Follow the approved plan strictly. Only modify directly related files. Explain every modified file.
```

## Prompt 3 — Review
```text
Review all changes for architecture, type safety, scope creep, hidden bugs, and missing validations.
```

## Prompt 4 — Test
```text
Generate test cases and manual QA checklist for this task based on unit-test-checklist.md.
```

## Prompt 5 — UI sang task
```text
Analyze this UI/image and convert it into a task file using TASK_TEMPLATE.ts. Include goal, requirements, business rules, related files, risks, and acceptance criteria.
```

# 📘 11. TASK TEMPLATE nâng cấp cho Level 2

Ngoài các mục cũ, task Level 2 nên thêm:

- `taskSource`: screenshot | text | bug report | spec
- `affectedLayers`: validator | repository | service | api | ui
- `riskLevel`: low | medium | high
- `rollbackPlan`
- `regressionTargets`
- `manualQaChecklist`

Ví dụ:

```ts
export const TASK_CONTROL = {
  taskSource: 'screenshot',
  affectedLayers: ['repository', 'service', 'ui'],
  riskLevel: 'medium',
  rollbackPlan: 'Revert menu filter changes if list query breaks current behavior',
  regressionTargets: ['existing search', 'status filter', 'menu table rendering'],
  manualQaChecklist: [
    'lọc theo menu type',
    'search + filter kết hợp',
    'click action link đúng màn hình',
  ],
};
```
